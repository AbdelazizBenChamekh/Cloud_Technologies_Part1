##### 3.3. Методология классификации
Для каждой строки исходного файла выполнялись следующие шаги:

Шаг 1: Определение Product Code → преобразование в понятный Service Type.
Пример: AmazonRedshift → Amazon Redshift.
Источник: Официальное наименование в AWS Console.

Шаг 2: Выбор IT Tower на основе типа сервиса.
Пример:

Amazon S3 → Storage (согласно категоризации AWS)

AWS Directory Service → Cloud Services

Amazon Redshift → Cloud Services (исследование показало, что Redshift относится к аналитическим сервисам, а не чистым базам данных)

Шаг 3: Выбор Service Family согласно таблице соответствий.
Пример:

Storage → Storage&Content Delivery

Cloud Services → Analytics, Application Services, Security and Identity и т.д.

Шаг 4: Анализ Usage Type для определения подтипа и типа использования.
Пример:

%Node:ra% → Node (подтип), Node Usage (тип использования)

%EarlyDelete-GDA → Glacier Deep Archive (подтип), Early Delete (тип использования)
Примечание: Для понимания сокращений (GDA, SIA, ZIA) потребовалось обращение к документации S3 Storage Classes.

Шаг 5: Заполнение всех пяти столбцов классификации.

#### 7. Анализ сервисов AWS и их российские аналоги
В процессе классификации я изучил несколько ключевых сервисов AWS. Вот что я узнал:

##### 7.1. Amazon Redshift
Что это: Облачное хранилище данных (data warehouse) для аналитики.
Зачем нужно: Компании используют Redshift для хранения больших объемов данных и быстрого выполнения аналитических запросов. Например, для отчетов по продажам, анализа поведения пользователей.
Особенность: Отличается от обычных баз данных (как RDS) — оптимизирован именно для аналитических нагрузок, а не для транзакций.
Российский аналог:

Яндекс.Облако: ClickHouse (Managed Service for ClickHouse) — тоже колоночное хранилище для аналитики

VK Cloud: Greenplum — распределенное хранилище данных

Отличие: В российских облаках обычно проще тарификация, но меньше автоматических функций

##### 7.2. Amazon S3 (Simple Storage Service)
Что это: Основное объектное хранилище AWS.
Зачем нужно: Хранение любых файлов — от картинок на сайте до резервных копий баз данных.
Что удивило: Столько классов хранения! Standard для частого доступа, Glacier для архивов, Intelligent-Tiering для автоматической оптимизации. В биллинге это видно по разным строчкам: EarlyDelete-GDA (Glacier Deep Archive), EarlyDelete-SIA (Standard-Infrequent Access).
Российский аналог:

Яндекс.Облако: Object Storage

VK Cloud: Объектное хранилище

Важное отличие: У российских провайдеров обычно меньше классов хранения (часто только "горячее" и "холодное")

##### 7.3. AWS Directory Service
Что это: Управляемый сервис Active Directory в облаке.
Зачем нужно: Компаниям, которые хотят использовать Windows-сервисы в AWS (например, файловые серверы, приложения, требующие Active Directory).
Интересно: В биллинге есть разные варианты — MicrosoftAD (полноценный AD), SimpleAD (упрощенный), AD Connector (для связи с локальным AD).
Российский аналог:

VK Cloud: Managed Active Directory

Яндекс.Облако: Пока нет прямого аналога, рекомендуют использовать сторонние решения

##### 7.4. Amazon Glacier
Что это: Сверхдешевое архивное хранилище.
Зачем нужно: Хранение данных, которые почти никогда не нужны (юридические архивы, старые бэкапы, медиа-архивы).
Важный нюанс: Восстановление данных занимает от нескольких часов до суток, за срочное восстановление — доплата. В биллинге это видно по разным типам запросов (Requests-Tier1, Tier3).
Российский аналог:

Яндекс.Облако: "Холодное" хранилище в Object Storage

Selectel: Архивное хранилище

##### 7.5. Amazon SNS (Simple Notification Service)
Что это: Сервис отправки уведомлений.
Зачем нужно: Отправка SMS, email, push-уведомлений из приложений. Например, коды подтверждения для входа, уведомления о заказах.
В биллинге видно: Разные типы отправки — SMS (плата за отправку), HTTP/SQS доставка (плата за попытки доставки).
Российский аналог:

Яндекс.Облако: Message Queue (для части функций)

VK Cloud: Notification Service

Чаще в России: Используют отдельные сервисы типа SMS.ru, UniSender

#### 9. Личный опыт и сложности
##### 9.1. Что было самым сложным
Расшифровка сокращений — без документации AWS невозможно было понять, что такое "GDA", "SIA", "ZIA" в S3. Пришлось искать в официальной документации.

Определение категорий — не всегда очевидно, куда отнести сервис. Redshift — это "Database" или "Analytics"? После изучения решил, что "Analytics", так как это data warehouse.

Налоги и дополнительные сборы — отдельные строки "Tax%" в биллинге. В российских облаках обычно цена "все включено".

##### 9.2. Что удивило
Детализация биллинга — AWS показывает каждую мелочь: разные типы запросов к S3, разные классы хранения.

Разные модели оплаты — например, в SNS плата за отправку SMS и отдельно за попытки доставки.

Архитектурные различия — то, что в AWS несколько отдельных сервисов, в российских облаках часто объединено в один.

##### 9.3. Практическая польза от работы
Теперь, если мы будем работать в компании, использующей AWS:

Сможем читать биллинг и понимать, за что именно платим

Сможем предлагать оптимизации (например, перенос старых данных в Glacier)

Понимаем разницу между сервисами и могу выбирать подходящие аналоги в российских облаках

#### 10. Сравнение с российскими облаками (личные выводы)
##### 10.1. Структура биллинга
AWS: Очень детализированный, много отдельных строк, сложно для новичка
Российские облака: Проще, понятнее, но меньше возможностей для детального анализа

##### 10.2. Набор сервисов
AWS: Огромный выбор, для каждой задачи свой специализированный сервис
Российские облака: Основные сервисы есть, но меньше специализированных options

##### 10.3. Стоимость и оптимизация
AWS: Можно сильно оптимизировать, но нужно глубоко разбираться
Российские облака: Меньше вариантов оптимизации, но и меньше шансов ошибиться

#### 11. Рекомендации для компаний
При переходе с AWS на российское облако:

S3 → Объектное хранилище (Яндекс, VK)

Redshift → ClickHouse (Яндекс) или Greenplum (VK)

Directory Service → Managed AD (VK) или кастомное решение

Для оптимизации затрат в AWS:

Анализировать биллинг по категориям (IT Tower)

Смотреть на типы использования (Usage Type)

Переносить редко используемые данные в Glacier

Для начинающих:

Начинать с российских облаков — проще разобраться

Изучать AWS уже после понимания основ

Мои главные выводы из этой работы:

Облачные сервисы — это не магия, а вполне логичная структура

Биллинг показывает не только "сколько платим", но и "как используем"

Российские облака догоняют AWS по функционалу, но еще есть куда расти

Самое важное — понимать, зачем нужен каждый сервис, а не просто знать его название
